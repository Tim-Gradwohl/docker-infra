QBITTORRENTVPN STACK (GLUETUN + QBITTORRENT + PIA WIREGUARD)
====================================================================

Purpose:
- Run qBittorrent strictly behind a PIA WireGuard tunnel using Gluetun.
- Use PIA Next-Gen port forwarding.
- Keep qBittorrent’s incoming port aligned automatically.
- Self-heal dead WireGuard tunnels via automated regeneration.

--------------------------------------------------------------------
CONTAINERS
--------------------------------------------------------------------

- qbittorrentvpn_gluetun
- qbittorrentvpn-qbittorrent-1
- qbittorrentvpn_pia-pf
- qbittorrentvpn_pf-writer
- qbittorrentvpn_port-sync

--------------------------------------------------------------------
CORE ENFORCEMENT ("ALWAYS VPN")
--------------------------------------------------------------------

- qBittorrent uses: network_mode: service:gluetun
- All traffic shares Gluetun’s network namespace.
- If Gluetun is down, qBittorrent loses network connectivity.

--------------------------------------------------------------------
WIREGUARD MODEL
--------------------------------------------------------------------

- Gluetun runs in custom WireGuard mode.
- wg0.conf is generated externally and bind-mounted:

  /home/tim/stacks/apps/qbittorrentvpn/wireguard-pia/wg0.conf
    ->
  /gluetun/wireguard/wg0.conf (read-only)

- WireGuard can report "setup complete" while passing no traffic.
- Dead/stale endpoints are a known failure mode and require regeneration.

--------------------------------------------------------------------
PORT FORWARDING (PIA NEXT-GEN)
--------------------------------------------------------------------

- Handled by thrnz/docker-wireguard-pia (pia-pf container).
- Forwarded port written to: /pia-shared/port.dat

--------------------------------------------------------------------
PORT PROPAGATION PIPELINE
--------------------------------------------------------------------

1) pia-pf writes port.dat
2) pf-writer mirrors to /gluetun/forwarded_port
3) port-sync reconciles qBittorrent configuration
4) qBittorrent is restarted only when port changes

--------------------------------------------------------------------
AUTOMATIC PORT SYNC (QBITTORRENT)
--------------------------------------------------------------------

- port-sync uses docker:cli image.
- Mounts:
  - /var/run/docker.sock
  - qBittorrent config directory (read-write)
  - /gluetun/forwarded_port

- Updates:
  - Session
  - Connection

- Persists last applied port in:
  /config/.last_forwarded_port

--------------------------------------------------------------------
SECURITY NOTE
--------------------------------------------------------------------

- port-sync mounts Docker socket.
- Treat this container as high-trust.
- Scripts are versioned under:
  apps/qbittorrentvpn/scripts/

--------------------------------------------------------------------
KNOWN FAILURE MODE: DEAD WIREGUARD TUNNEL
--------------------------------------------------------------------

Symptoms:
- qbittorrentvpn_gluetun becomes unhealthy
- Logs show:
  - "i/o timeout"
  - "failed to pass the healthcheck"
  - repeated VPN restarts
- WireGuard reports "setup complete" but no traffic flows
- Inside container:
  ping 1.1.1.1 → fails

Root cause:
- Stale or non-functional PIA endpoint in wg0.conf

--------------------------------------------------------------------
MANUAL RECOVERY PROCEDURE
--------------------------------------------------------------------

1) Stop stack:
   stk down qbittorrentvpn

2) Remove old config:
   rm wireguard-pia/wg0.conf

3) Regenerate using PIA generator container

4) Copy:
   /etc/wireguard/wg0.conf → wireguard-pia/wg0.conf

5) Start stack:
   stk up qbittorrentvpn

6) Validate:
   docker exec qbittorrentvpn_gluetun ping -c1 1.1.1.1

--------------------------------------------------------------------
AUTOMATED RECOVERY
--------------------------------------------------------------------

Two host-level tools handle recovery:

1) qb-vpn-refresh  (repair primitive)
2) qb-vpn-watchdog (detector + orchestrator)

Both are located in:
~/stacks/bin/

--------------------------------------------------------------------
qb-vpn-refresh (REPAIR COMMAND)
--------------------------------------------------------------------

Purpose:
- Deterministic recovery of dead WireGuard tunnel

Steps:
1) Backup stack
2) Stop stack
3) Remove wg0.conf
4) Generate new config via PIA container
5) Copy wg0.conf from generator container
6) Restart stack

Usage:
  qb-vpn-refresh

Notes:
- Uses credentials from:
  ~/stacks/shared/.env.secrets
- Does NOT perform detection
- Does NOT perform long-term health verification

--------------------------------------------------------------------
qb-vpn-watchdog (SELF-HEALING LOOP)
--------------------------------------------------------------------

Purpose:
- Detect dead tunnel condition
- Trigger qb-vpn-refresh automatically
- Verify recovery

Detection conditions (all required):
- Gluetun container is unhealthy
- Logs match dead-tunnel pattern
- Raw IP connectivity fails:
  ping 1.1.1.1

Behavior:
- Runs periodically (cron)
- Applies cooldown (default: 30 minutes)
- Prevents concurrent runs via lock
- Calls qb-vpn-refresh
- Verifies:
  - raw IP connectivity restored
  - DNS connectivity
  - container health

State + logs:
  ~/stacks/state/qbittorrentvpn/

--------------------------------------------------------------------
CRON CONFIGURATION (WSL)
--------------------------------------------------------------------

Run every 5 minutes:

  */5 * * * * /home/tim/stacks/bin/qb-vpn-watchdog >/dev/null 2>&1

Start cron manually in WSL:

  sudo service cron start

KEY INSIGHT
--------------------------------------------------------------------

WireGuard may appear "connected" while passing zero traffic.

TARGET FLOW EXPLANATION (3.9.51)
====================================================================

This section formalizes the intended traffic model after HTTPS production migration.

### LAN (DNS Challenge via Cloudflare + Let's Encrypt Production)

Client (LAN) 
→ AdGuard DNS 
→ photos.timopoly.com resolves to 192.168.188.23 
→ Traefik (:443 websecure) 
→ Target Service (Docker container)

Explanation:

- AdGuard performs split-horizon DNS by rewriting *.${LAN_DOMAIN} or production hostnames directly to the Windows host LAN IP.
- LAN clients never traverse Cloudflare or the public internet.
- Traefik serves a valid public Let's Encrypt certificate even for LAN clients.
- ACME validation occurs via DNS-01 (Cloudflare), not HTTP challenge, so no inbound port exposure is required.
- This guarantees identical hostname usage for LAN and WAN without certificate mismatch.

### WAN (Public Access via Cloudflare Tunnel)

Client (Internet) 
→ Cloudflare DNS 
→ Cloudflare Edge 
→ Cloudflare Tunnel 
→ cloudflared (Docker) 
→ Traefik (:443 websecure) 
→ Target Service (Docker container)

Explanation:

- Public DNS is authoritative at Cloudflare.
- No inbound ports are required on the router when using the tunnel model.
- cloudflared establishes an outbound-only encrypted tunnel to Cloudflare.
- Cloudflare forwards traffic through the tunnel to Traefik.
- Traefik performs host-based routing and TLS termination (valid production certificate).
- Authentik ForwardAuth is evaluated before request reaches the service (if middleware enabled).

This architecture represents proper split-horizon DNS:

- Internal DNS overrides ensure LAN clients use the shortest possible path.
- External clients traverse Cloudflare edge infrastructure.
- Both flows converge at Traefik for unified routing and authentication logic.

====================================================================
YT2MIDI_V3 STACK (NEW IN 3.9.54)
====================================================================

Purpose:
- Next-generation YouTube → MP3 + MIDI pipeline with improved separation
  of web UI and backend processing.
- Designed to keep UI lightweight while heavy processing runs in a
  dedicated CUDA-enabled API container.

Containers:
- yt2midi_v3-ui
- yt2midi_v3-api

Architecture:
- UI container:
    nginx:alpine serving static web interface.
- API container:
    FastAPI backend using CUDA runtime image for GPU acceleration.

Networks:
- internal (UI ↔ API communication)
- proxy (Traefik reverse proxy network)

Traefik Routing:
- traefik.enable=true
- traefik.docker.network=proxy
- Host(yt2midi-v3.${BASE_DOMAIN})
- EntryPoint: websecure
- TLS enabled (Cloudflare DNS resolver)
- Backend service port: 80 (nginx UI)

Access (current):
- LAN / WAN: https://yt2midi-v3.${BASE_DOMAIN}

Storage:
- /home/tim/stacks/apps/yt2midi_v3/data/in -> /data/in
- /home/tim/stacks/apps/yt2midi_v3/data/out -> /data/out

Git Hygiene:
- apps/yt2midi_v3/data/ ignored in Git (runtime artifacts only).
